2. PROBLEMA #1: MISMATCH CRÍTICO DE CLAVES SUPABASE
================================================================================

SEVERIDAD: 🔴 CRÍTICO - BLOQUEANTE TOTAL

DESCRIPCIÓN:
El frontend y las Edge Functions usan DIFERENTES claves de Supabase, causando
que TODAS las validaciones JWT fallen.

EVIDENCIA TÉCNICA:

📁 Archivo: .env
---

VITE_SUPABASE_PUBLISHABLE_KEY="sb_publishable_FGi-YziQ6se1k5E8QeHJJw_e3_1jbBZ"
VITE_SUPABASE_URL="https://xypbohvarofufrdkfeaj.supabase.co"
---

📁 Archivo: src/lib/supabase.ts (línea 11)
---

---
✅ Frontend usa: sb_publishable_FGi-YziQ6se1k5E8QeHJJw_e3_1jbBZ (NUEVA CLAVE)

📁 Archivo: supabase/functions/gmail-auth-url/index.ts (línea 18)
---
const anonKey = Deno.env.get("SUPABASE_ANON_KEY") ?? "";
---
❌ Edge Function espera: SUPABASE_ANON_KEY (CLAVE LEGACY)

📁 Archivo: supabase/functions/gmail-connection-status/index.ts (línea 13)
---

---
❌ Edge Function espera: SUPABASE_ANON_KEY (CLAVE LEGACY)

ANÁLISIS DEL PROBLEMA:

1. El frontend genera JWT usando la clave PUBLISHABLE (sb_publishable_*)
2. Las Edge Functions intentan validar el JWT usando ANON_KEY (diferente)
3. Supabase rechaza la validación porque las claves no coinciden
4. Resultado: 401 Unauthorized / Invalid JWT

FLUJO DEL ERROR:

Frontend (React)
  ↓ Genera JWT con sb_publishable_FGi-YziQ6se1k5E8QeHJJw_e3_1jbBZ
  ↓ Envía: Authorization: Bearer <jwt>
  ↓
Edge Function (gmail-auth-url)
  ↓ Lee: Deno.env.get("SUPABASE_ANON_KEY")
  ↓ Intenta validar JWT con clave DIFERENTE
  ↓ Llama: ${supabaseUrl}/auth/v1/user
  ↓
Supabase Auth API
  ↓ Compara JWT signature con ANON_KEY
  ❌ FALLA: Las claves no coinciden
  ↓ Responde: 401 Unauthorized

POR QUÉ LOS INTENTOS ANTERIORES FALLARON:
- Se corrigió la lógica de validación JWT (✅ correcto)
- Se agregaron logs de debugging (✅ correcto)
- PERO NUNCA se verificó que las claves coincidieran (❌ error)

SOLUCIÓN REQUERIDA:

Opción A (RECOMENDADA): Usar la misma clave en todos lados
---
1. Verificar qué clave está configurada en Supabase Dashboard
2. Si es sb_publishable_*, actualizar Edge Functions:
   - Cambiar SUPABASE_ANON_KEY → SUPABASE_PUBLISHABLE_KEY
   - O crear secret SUPABASE_ANON_KEY con el valor de publishable
3. Si es una clave legacy (eyJ...), actualizar frontend

Opción B: Usar Service Role Key en Edge Functions
---
1. Las Edge Functions pueden usar SERVICE_ROLE_KEY para validar
2. Esto es más seguro para operaciones backend
3. Requiere cambiar la lógica de validación

ARCHIVOS A MODIFICAR:
- supabase/functions/gmail-auth-url/index.ts (línea 18)

================================================================================

SEVERIDAD: 🟠 ALTA - CAUSA PROBLEMAS INTERMITENTES

DESCRIPCIÓN:
El sistema mezcla múltiples patrones de autenticación de forma inconsistente,
causando confusión y potenciales fallos.

PATRONES DETECTADOS:

Patrón A: Validación JWT Manual (gmail-auth-url, gmail-connection-status)
---
async function getUserFromJwt(authHeader: string) {
  const supabaseUrl = Deno.env.get("SUPABASE_URL") ?? "";
  const anonKey = Deno.env.get("SUPABASE_ANON_KEY") ?? "";
  
  const r = await fetch(`${supabaseUrl}/auth/v1/user`, {
    method: "GET",

      Authorization: authHeader,
      apikey: anonKey,
    },
  });
  // ...
}

❌ Desventaja: Código duplicado, propenso a errores

Patrón B: Supabase Client con Auth Header (gmail-send)
---
const supabaseClient = createClient(
  Deno.env.get('SUPABASE_URL') ?? '',
  Deno.env.get('SUPABASE_ANON_KEY') ?? '',
  {
    global: {

    },
  }

---
✅ Ventaja: Usa SDK oficial
❌ Desventaja: Requiere ANON_KEY correcta

Patrón C: Service Role (gmail-callback)
---

  Deno.env.get('SUPABASE_URL') ?? '',
  Deno.env.get('SUPABASE_SERVICE_ROLE_KEY') ?? ''

---
✅ Ventaja: Bypass RLS, operaciones admin
❌ Desventaja: Peligroso si se expone

INCONSISTENCIAS DETECTADAS:

1. Nombres de variables inconsistentes:
   - getUserFromJwt() en gmail-auth-url
   - getUserFromJwt() en gmail-connection-status (duplicado)
   - supabaseClient.auth.getUser() en gmail-send
   - No validación en gmail-callback (usa service role)

2. Manejo de errores inconsistente:
   - gmail-auth-url: Retorna { user, error }
   - gmail-send: Lanza excepción
   - gmail-callback: Redirige con error

3. Headers requeridos inconsistentes:
   - Algunas funciones requieren Authorization + apikey
   - Otras solo Authorization
   - gmail-callback no requiere auth (es callback público)

RECOMENDACIÓN ARQUITECTÓNICA:

Crear un módulo compartido de autenticación:

📁 supabase/functions/_shared/auth.ts
---
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2.39.3';

export async function authenticateRequest(req: Request) {
  const authHeader = req.headers.get('Authorization');
  if (!authHeader) {
    throw new Error('Missing Authorization header');
  }

  const supabaseClient = createClient(
    Deno.env.get('SUPABASE_URL') ?? '',
    Deno.env.get('SUPABASE_ANON_KEY') ?? '',
    {
      global: {
        headers: { Authorization: authHeader },
      },
    }
  );

  const { data: { user }, error } = await supabaseClient.auth.getUser();
  
  if (error || !user) {
    throw new Error('Invalid or expired token');
  }

  return { user, supabaseClient };
}
---

Luego usar en todas las funciones:
---
import { authenticateRequest } from '../_shared/auth.ts';

Deno.serve(async (req) => {
  try {
    const { user, supabaseClient } = await authenticateRequest(req);
    // ... resto de la lógica
  } catch (error) {
    return new Response(JSON.stringify({ error: error.message }), {
      status: 401,
      headers: corsHeaders,
    });
  }
});
---

BENEFICIOS:
- Código DRY (Don't Repeat Yourself)
- Manejo de errores consistente
- Fácil de actualizar si cambia la lógica de auth
- Menos propenso a bugs

SEVERIDAD: 🟡 MEDIA - FUNCIONA PERO PUEDE MEJORAR

DESCRIPCIÓN:
El servicio gmailAccountService.ts tiene lógica compleja para extraer

EVIDENCIA TÉCNICA:

---

PROBLEMAS:

1. Uso de (supabase as any) - Bypass de TypeScript

   - Indica incertidumbre sobre la API del cliente

3. anonKey puede ser undefined
   - Se loguea pero no se lanza error
   - Puede causar fallos silenciosos

BENEFICIOS:
- Errores claros en los logs de Supabase
- Fácil diagnóstico de problemas de configuración
- No requiere hacer requests para descubrir el problema
